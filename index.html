HTTP协议

[TOC]
HyperText Transform Protocol 超文本转移协议,现在约定俗成的说法为超文本传输协议.
所谓协议，就是双方沟通方式的一种规则，例如：怎么知道通目标？由哪一边发起通信？通信采用的语言是什么？等等

HTTP是用来干嘛的？

HTTP是应用层通信协议，其最初目的是用来共享传输文本的。

那要构建这种通信技术需要哪些基本素材呢？

  1. 通信内容： 基于SGML(Standard Generalized Markup Language,标准通用语言)作为页面文本标记语言的的HTML（超文本标记语言）
  2. 通信协议： 采用什么网络协议–HTTP协议
  3. 通讯地址： 采用URL（Uniform Resouce Location）方式(URI的子集)

HTTP是TCP/IP协议族的一个子集，HTTP在TCP/IP中是怎么样的地位或作用？

TCP/IP是互联网相关连协议集合的总称。
TCP/IP协议族最重要的一点是分层（类似于软件设计中的SoC原则，分离关注点），分为4层：

  1. 应用层： 主要负责与用户相关的通信活动。例如：HTTP,FTP(File Transfer Protocol)
  2. 传输层： 提供网络中两台计算机之间的数据传输。例如：TCP（Transmission Control Protocol）,UDP(User Data Protocol)
  3. 网络层: 处理网络上流动的的数据包，主要作用是规定了通过怎样的传输线路到达对方计算机，即把数据包发送给对方。所谓数据包是网络传输的最小单位。例如：IP(Internet Protocol)
  4. 链路层（数据链路层，网络接口层）： 处理连接网络的物理硬件部分。例如:MAC

整个通信流程是什么样的呢？

基于TCP/IP协议族进行通信时，会按分层顺序与对方通信。发送端从应用层往下走，接收端往应用层上走。
重点在于理解每个层的作用，于是才能理解其对应采用通信协议是干嘛的！
例如：

  1. 当某应用发出一个HTTP请求
  2. 传输层接收到应用层的数据，为传输方便，将接收到的数据即HTTP的请求报文进行分割，并在每个报文打上标记序号及端口，然后转发给网络层
  3. 网络层接受到报文后，增加了作为通信目的地的MAC(Media Access Control Address)地址(通过ARP协议)后转发给链路层
  4. 通信的请求准备完毕，发往网络
  5. 接收端在链路层接收到数据，然后按序网上发送，直到应用层。

注：

  • 发送端在每一层都会打上一个属于该层的首部(也称报文)；而接收端则会在每一层将与之对应层的报文解开
  • 网络层IP协议的作用是将数据发送给对方，但这依赖于两个重要条件：IP地址和MAC地址
    • IP地址用于指明节点被分配到的地址（可变）
    • MAC地址是网卡硬件的固件地址（基本不变）
  • 在网络中的通信通常有两种情况：同网段和非同网段（同一局域网）。更常见的是非同网段，那么会经过多台计算机或者网络设备才能连接到对方。然而在中转的过程中，是依赖于搜索MAC地址来寻找下一个中转目标（机制如此）。但是在最初请求时只会给出IP地址，这就依赖于ARP（Address Resolution Protocol)通过IP地址反查出MAC地址。

TCP协议

TCP协议位于传输层，用于提供可靠的字节流服务。

  • 字节流服务(Byte Stream Service)是指,为了传输方便将大块的数据分割成以报文段(segment)为单位的数据包进行管理。
  • 可靠，是指能够将数据准确可靠的传给对方
    即为了容易传输大文件采用分割方案，又有三次握手(three-way handshaking)策略来保证数据的传输的准确无误。

DNS服务

DNS服务位于应用层，提供域名到IP地址的解析服务。
域名或主机名是IP地址的别称，其作用是方便记忆。

HTTP协议

HTTP能够区别客户端和服务器。因为HTTP是由客户端发起请求，服务器端相应该请求；一定是从客户端发起，服务器端接受到请求后才会响应。

无状态的HTTP协议

HTTP协议是无状态（stateless）协议,也就是说HTTP协议自身是不对请求和响应之间的状态进行保存的，不作持久化处理。

  • 优点
    • 能够快速处理大量事物
    • 协议可伸缩性强

  • 随着Web发展而带来的问题
    当业务存在与用户交互时，就存在着对用户信息的保存，而HTTP协议本身是无状态协议，因此无法实现这部分功能。于是便引入了Cookie技术来管理状态。

  • 使用Cookie进行状态管理
    原理： Cookie会自动根据响应报文中Set-Cookie首部字段，来告诉客户端保存Cookie。之后再与该服务器发生请求时，客户端会自动在请求报文中加入Cookie值后发送出去。
    例如：如下过程

  1. 首先客户端发起请求

  2. 服务器端响应，并在首部加入Set-Cookie

  3. 再次发出向该服务器发出请求时,会带上Cookie

HTTP协议常用的方法（注意大小写）

  1. GET:获取资源
    GET方法用于请求已经被URI识别的资源。然后指定的资源经过服务器解析后返回响应内容。

  2. POST:传输实体主体
    POST方法用来传输实体主体。虽然GET方法也可以，但通常不用。POST方法的虽然也可以获取相应的主体内容，但是其主要目的是用来传输实体主体的

  3. PUT:传输文件
    PUT方法用来传输文件。在请求报文的主体中包含文件，然后会保存到请求的URI指定的位置。

  4. DELETE：删除文件
    DELETE方法用来删除文件。按请求URI删除指定的资源。

持久连接

早期每进行一次HTTP通信，都需要与TCP建立连接，而TCP的连接需要三次握手，当请求量小的时候还没问题，当请求量大的时候，这就是一种损耗了。

  • 解决方案： 持久连接（HTTP Persistent Connections或HTTP keep-alive）:即只要任意一端没有明确提出断开连接，那么就会保持TCP连接状态。（HTTP/1.1所有连接默认都是持久连接）
    • 优点： 节省额外开销，降低服务器的负载。页面加载也加快了，因为省下来额外的建立时间。
